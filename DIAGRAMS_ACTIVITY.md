# دياجرامات Activity - شرح مفصل للأدوار

## مقدمة
هذا الملف يحتوي على شرح مفصل لدياجرامات Activity لكل رول في النظام. تم اختيار الدياجرامات الأساسية والأكثر أهمية لكل رول، مع شرح تدفق العمليات خطوة بخطوة مع جميع الحالات والشروط والفلترة.

---

## 1. Artist (الفنان)

### 1.1 رفع Artwork جديد - تدفق العملية

- الفنان يبدأ العملية بالدخول إلى صفحة إنشاء عمل فني جديد
- الفنان يملأ النموذج: يدخل العنوان (مطلوب)، يدخل الوصف (اختياري)، يختار الفئة من القائمة (مطلوب)، يرفع الملف (اختياري)
- الفنان يضغط على زر "حفظ"
- النظام يبدأ التحقق من البيانات: يتحقق من وجود العنوان، يتحقق من أن الفئة موجودة في قاعدة البيانات، يتحقق من نوع الملف إذا تم رفعه
- إذا فشل التحقق: النظام يعرض رسالة خطأ، الفنان يعود لإصلاح البيانات، العملية تعود إلى خطوة ملء النموذج
- إذا نجح التحقق: النظام يحفظ الملف في المجلد المخصص (إذا تم رفع ملف)
- النظام يحسب ضريبة المنصة تلقائياً بناءً على نوع الفئة (موسيقى: 100 دج، صورة: 200 دج، فيديو: 500 دج)
- النظام ينشئ سجل جديد في قاعدة البيانات للعمل الفني بحالة "PENDING"
- النظام يُرسل إشعار لجميع Gestionnaires في وكالة الفنان بوجود عمل فني جديد للمراجعة
- النظام يُعيد توجيه الفنان لصفحة الأعمال قيد الانتظار مع رسالة نجاح
- العملية تنتهي بنجاح

### 1.2 دفع ضريبة المنصة - تدفق العملية مع الشروط

- الفنان يبدأ العملية بفتح عمل فني معتمد من قائمة أعماله
- الفنان يرى أن العمل معتمد ولكن يحتاج لدفع ضريبة المنصة
- الفنان يضغط على زر "دفع ضريبة المنصة"
- النظام يتحقق من حالة العمل: هل الحالة هي "APPROVED"؟
- إذا لم تكن الحالة "APPROVED": النظام يعرض رسالة خطأ "يجب أن يكون العمل معتمداً قبل دفع الضريبة"، العملية تنتهي
- إذا كانت الحالة "APPROVED": النظام يتحقق من حالة ضريبة المنصة: هل دُفعت بالفعل؟
- إذا دُفعت الضريبة بالفعل: النظام يعرض رسالة خطأ "تم دفع ضريبة المنصة بالفعل"، العملية تنتهي
- إذا لم تكن مدفوعة: النظام يحصل على محفظة الفنان (أو ينشئها إذا لم تكن موجودة)
- النظام يتحقق من رصيد المحفظة: هل الرصيد >= ضريبة المنصة؟
- إذا كان الرصيد غير كافٍ: النظام يعرض رسالة خطأ "رصيد المحفظة غير كافٍ، يرجى شحن المحفظة"، العملية تنتهي
- إذا كان الرصيد كافياً: النظام يبدأ معاملة مالية (Transaction)
  - النظام يخصم المبلغ من محفظة الفنان
  - النظام يحصل على محفظة الوكالة (أو ينشئها إذا لم تكن موجودة)
  - النظام يضيف المبلغ لمحفظة الوكالة
  - النظام ينشئ معاملة مالية للفنان (نوع: PLATFORM_TAX، المبلغ: سالب)
  - النظام ينشئ معاملة مالية للوكالة (نوع: PLATFORM_TAX، المبلغ: موجب)
  - النظام يحدث حالة العمل الفني: platform_tax_status = "PAID"
- النظام يُرسل إشعار للفنان بتفعيل العمل
- النظام يُرسل إشعار لـ Gestionnaire بدفع الضريبة
- النظام يُعيد توجيه الفنان للصفحة مع رسالة نجاح
- العملية تنتهي بنجاح

## 2. Agent (الوكيل)

### 2.0 تحديث حالة Mission وإنشاء PV من Mission - تدفق العملية

- Agent يبدأ العملية بالدخول إلى صفحة المهام (Missions)
- Agent يرى قائمة المهام المخصصة له (حالة: ASSIGNED, IN_PROGRESS, DONE)
- Agent يفتح مهمة معينة
- Agent يمكنه تحديث حالة المهمة (ASSIGNED → IN_PROGRESS → DONE)
- إذا غير Agent حالة المهمة:
  - Agent يختار الحالة الجديدة من القائمة
  - Agent يضغط على زر "تحديث الحالة"
  - النظام يحدث حالة المهمة
  - النظام يُرسل إشعار لـ Gestionnaire بتحديث حالة المهمة
- Agent يمكنه إنشاء PV مرتبط بهذه المهمة
- Agent يضغط على زر "إنشاء PV من هذه المهمة"
- Agent يُعاد توجيهه لصفحة إنشاء PV (والـ Mission مرتبط تلقائياً)
- Agent يملأ معلومات PV: اسم المتجر، نوع المتجر، تاريخ التفتيش، صور الأدلة
- Agent يضغط على زر "إنشاء"
- النظام يبدأ التحقق من البيانات
- إذا فشل التحقق: النظام يعرض رسالة خطأ، Agent يعود لإصلاح البيانات، العملية تعود إلى خطوة ملء النموذج
- إذا نجح التحقق: النظام يحفظ الصور في المجلد المخصص
- النظام ينشئ PV جديد بحالة "OPEN" ويربطه بالـ Mission
- النظام يغير حالة المهمة تلقائياً إلى "IN_PROGRESS" (إذا لم تكن بالفعل)
- النظام يُرسل إشعار لـ Gestionnaire بإنشاء PV من المهمة
- النظام يُعيد توجيه Agent لصفحة PV الجديد مع رسالة نجاح
- العملية تنتهي بنجاح
- لاحقاً، بعد إغلاق PV: النظام يغير حالة المهمة تلقائياً إلى "DONE"

### 2.1 فتح PV جديد - تدفق العملية

- الوكيل يبدأ العملية بالدخول إلى صفحة إنشاء PV جديد
- الوكيل يختار مهمة (Mission) موجودة (اختياري) من القائمة
- الوكيل يملأ النموذج: يدخل اسم المتجر (مطلوب)، يدخل نوع المتجر (مطلوب)، يختار تاريخ التفتيش (مطلوب)، يدخل ملاحظات (اختياري)، يرفع صور الأدلة (حتى 25 صورة)
- الوكيل يضغط على زر "إنشاء"
- النظام يبدأ التحقق من البيانات: يتحقق من وجود اسم المتجر، يتحقق من وجود نوع المتجر، يتحقق من وجود تاريخ التفتيش، يتحقق من الصور
- إذا فشل التحقق: النظام يعرض رسالة خطأ، الوكيل يعود لإصلاح البيانات، العملية تعود إلى خطوة ملء النموذج
- إذا نجح التحقق: النظام يحفظ الصور في المجلد المخصص
- النظام ينشئ PV جديد بحالة "OPEN"
- إذا كان مرتبطاً بمهمة: النظام يغير حالة المهمة إلى "IN_PROGRESS"
- النظام يُرسل إشعار لجميع Gestionnaires في وكالة الوكيل بفتح PV جديد
- النظام يُعيد توجيه الوكيل لصفحة PV الجديد مع رسالة نجاح
- العملية تنتهي بنجاح

### 2.2 إضافة Artwork إلى PV - تدفق العملية مع الشروط

- الوكيل يبدأ العملية بفتح PV موجود بحالة "OPEN"
- الوكيل يضغط على زر "إضافة عمل فني"
- الوكيل يختار الوكالة من القائمة
- النظام يُحمل قائمة الفنانين في الوكالة المختارة (فقط الفنانين المعتمدين: status = APPROVED) ويعرضها للوكيل
- الوكيل يختار الفنان من القائمة
- النظام يُحمل قائمة أعمال الفنان (فقط الأعمال المعتمدة ومدفوعة الضريبة: status = APPROVED AND platform_tax_status = PAID) ويعرضها للوكيل
- الوكيل يختار العمل الفني من القائمة
- النظام يتحقق من أن العمل الفني معتمد ومدفوع الضريبة
- إذا لم يكن معتمداً أو لم تدفع الضريبة: النظام يعرض رسالة خطأ "هذا العمل الفني لا يمكن استخدامه في PVs حتى يتم اعتماده ودفع الضريبة"، الوكيل يعود لاختيار عمل فني آخر، العملية تعود إلى خطوة اختيار العمل الفني
- إذا كان كل شيء صحيح: الوكيل يختار الجهاز (اختياري) من قائمة الأجهزة في PV
- الوكيل يختار طريقة الحساب (ساعات أو عدد)
- إذا اختار "ساعات": الوكيل يدخل عدد الساعات (على الأقل 0.5 ساعة)
- إذا اختار "عدد": الوكيل يدخل العدد (على الأقل 1)
- الوكيل يضغط على زر "إضافة"
- النظام يحسب الغرامة: (معامل الفئة × معامل الجهاز × الساعات/العدد × السعر الأساسي)
- النظام ينشئ سجل استخدام العمل الفني في PV (PVArtwork)
- النظام يحدث المبلغ الإجمالي للـ PV
- النظام يحدث المبلغ الإجمالي للجهاز (إذا كان مرتبطاً بجهاز)
- النظام يُرسل إشعار للفنان بأن عمله الفني تم استخدامه في PV
- النظام يُعيد توجيه الوكيل لصفحة PV مع رسالة نجاح
- العملية تنتهي بنجاح

### 2.3 إغلاق PV - تدفق العملية المعقدة

- الوكيل يبدأ العملية بإضافة جميع الأجهزة والأعمال الفنية للـ PV
- الوكيل يضغط على زر "تأكيد استلام الدفع"
- الوكيل يدخل المبلغ المستلم وطريقة الدفع (نقدي أو شيك)
- الوكيل يضغط على زر "تأكيد"
- النظام يتحقق من أن المبلغ المستلم يساوي المبلغ الإجمالي بالضبط (مع تسامح صغير للأخطاء العشرية)
- إذا لم يتطابق المبلغ: النظام يعرض رسالة خطأ "المبلغ المستلم يجب أن يساوي المبلغ الإجمالي بالضبط"، الوكيل يعود لإدخال المبلغ الصحيح، العملية تعود إلى خطوة إدخال المبلغ
- إذا تطابق المبلغ: النظام يحدث payment_method و cash_received_amount و agent_payment_confirmed = true
- النظام يُرسل إشعار لجميع Gestionnaires في وكالة الوكيل بتأكيد استلام الدفع
- بعد ذلك، الوكيل يضغط على زر "إغلاق PV"
- النظام يتحقق من أن PV ليس مغلقاً بالفعل
- إذا كان مغلقاً: النظام يعرض رسالة خطأ "هذا PV مغلق بالفعل"، العملية تنتهي
- إذا لم يكن مغلقاً: النظام يتحقق من أن الدفع تم تأكيده (agent_payment_confirmed = true)
- إذا لم يتم تأكيد الدفع: النظام يعرض رسالة خطأ "لا يمكن إغلاق PV: الدفع لم يتم تأكيده بعد"، العملية تنتهي
- إذا تم تأكيد الدفع: النظام يتحقق من أن المبلغ المستلم يساوي المبلغ الإجمالي بالضبط
- إذا لم يتطابق: النظام يعرض رسالة خطأ "المبلغ المستلم يجب أن يساوي المبلغ الإجمالي بالضبط"، العملية تنتهي
- إذا تطابق كل شيء: النظام يغير حالة PV إلى "CLOSED"
- إذا كان مرتبطاً بمهمة: النظام يغير حالة المهمة إلى "DONE"
- النظام يُرسل إشعار لجميع Gestionnaires في وكالة الوكيل بإغلاق PV
- النظام يُعيد توجيه الوكيل لصفحة PV مع رسالة نجاح
- العملية تنتهي بنجاح

## 3. Gestionnaire (المدير)

### 3.0 إنشاء Mission للـ Agent - تدفق العملية

- Gestionnaire يبدأ العملية بالدخول إلى صفحة المهام (Missions)
- Gestionnaire يضغط على زر "إنشاء مهمة جديدة"
- Gestionnaire يختار Agent من قائمة Agents في وكالته
- Gestionnaire يمكنه ربط المهمة بشكوى موجودة (اختياري)
- Gestionnaire يملأ النموذج: العنوان (مطلوب)، الوصف (اختياري)، الموقع (اختياري)، رابط الخريطة (اختياري)، التاريخ المقرر (اختياري)
- Gestionnaire يضغط على زر "إنشاء"
- النظام يبدأ التحقق من البيانات
- إذا فشل التحقق: النظام يعرض رسالة خطأ، Gestionnaire يعود لإصلاح البيانات، العملية تعود إلى خطوة ملء النموذج
- إذا نجح التحقق: النظام ينشئ مهمة جديدة بحالة "ASSIGNED"
- إذا كان مرتبطاً بشكوى: النظام يغير حالة الشكوى إلى "IN_PROGRESS" ويربطها بالمهمة
- النظام يُرسل إشعار للـ Agent بالمهمة الجديدة
- النظام يُعيد توجيه Gestionnaire لصفحة المهام مع رسالة نجاح
- العملية تنتهي بنجاح
- بعد ذلك، Agent يستقبل المهمة ويمكنه تحديث حالتها أو إنشاء PV منها

### 3.1 مراجعة وموافقة Artwork - تدفق العملية مع الشروط

- Gestionnaire يبدأ العملية بالدخول إلى صفحة الأعمال الفنية
- Gestionnaire يرى قائمة الأعمال الفنية قيد الانتظار من الفنانين في وكالته
- Gestionnaire يفتح عمل فني معين
- Gestionnaire يراجع الملف والمعلومات
- Gestionnaire يختار إما الموافقة أو الرفض
- إذا اختار الموافقة:
  - Gestionnaire يضغط على زر "موافقة"
  - النظام يغير حالة العمل إلى "APPROVED"
  - النظام يغير حالة ضريبة المنصة إلى "PENDING" (إذا لم تكن مدفوعة بالفعل)
  - النظام يحسب ضريبة المنصة (إذا لم تكن محسوبة)
  - النظام يُرسل إشعار للفنان بالموافقة وبضرورة دفع الضريبة لتفعيل العمل
  - النظام يُعيد توجيه Gestionnaire لصفحة الأعمال الفنية مع رسالة نجاح
  - العملية تنتهي بنجاح
- إذا اختار الرفض:
  - Gestionnaire يضغط على زر "رفض"
  - النظام يعرض حقل لإدخال سبب الرفض
  - Gestionnaire يدخل سبب الرفض (على الأقل 10 أحرف)
  - Gestionnaire يضغط على زر "تأكيد الرفض"
  - النظام يتحقق من أن سبب الرفض على الأقل 10 أحرف
  - إذا كان السبب أقل من 10 أحرف: النظام يعرض رسالة خطأ، Gestionnaire يعود لإدخال سبب أطول، العملية تعود إلى خطوة إدخال سبب الرفض
  - إذا كان السبب صحيحاً: النظام يغير حالة العمل إلى "REJECTED"
  - النظام يحفظ سبب الرفض
  - النظام يُرسل إشعار للفنان بالرفض مع السبب
  - النظام يُعيد توجيه Gestionnaire لصفحة الأعمال الفنية مع رسالة نجاح
  - العملية تنتهي بنجاح

### 3.2 التحقق من دفع PV وإضافته لمحفظة الوكالة - تدفق العملية المعقدة

- Gestionnaire يبدأ العملية بالدخول إلى صفحة PVs
- Gestionnaire يفتح PV مغلق (CLOSED) ومدفوع من Agent
- Gestionnaire يرى تفاصيل PV: المبلغ الإجمالي، المبلغ المستلم، طريقة الدفع
- Gestionnaire يضغط على زر "تأكيد الدفع"
- النظام يتحقق من أن Agent قد أكد استلام الدفع (agent_payment_confirmed = true)
- إذا لم يؤكد Agent: النظام يعرض رسالة خطأ "يجب أن يؤكد Agent استلام الدفع أولاً"، العملية تنتهي
- إذا أكد Agent: Gestionnaire يدخل المبلغ المستلم
- Gestionnaire يضغط على زر "تأكيد"
- النظام يتحقق من أن المبلغ المستلم يساوي المبلغ الإجمالي بالضبط
- إذا لم يتطابق المبلغ: النظام يعرض رسالة خطأ "المبلغ المستلم يجب أن يساوي المبلغ الإجمالي بالضبط"، Gestionnaire يعود لإدخال المبلغ الصحيح، العملية تعود إلى خطوة إدخال المبلغ
- النظام يتحقق من أن المبلغ المستلم يساوي المبلغ الذي أكده Agent بالضبط
- إذا لم يتطابق: النظام يعرض رسالة خطأ "المبلغ المستلم يجب أن يساوي المبلغ الذي أكده Agent بالضبط"، العملية تنتهي
- إذا تطابق كل شيء: النظام يتحقق من أن الدفع لم يتم تأكيده من قبل (payment_status != "VALIDATED")
- إذا تم تأكيد الدفع من قبل: النظام يعرض رسالة خطأ "تم تأكيد الدفع بالفعل"، العملية تنتهي
- إذا لم يتم التأكيد: النظام يبدأ معاملة مالية (Transaction)
  - النظام يحصل على محفظة الوكالة (أو ينشئها إذا لم تكن موجودة)
  - النظام يضيف المبلغ لمحفظة الوكالة
  - النظام يحدث حالة PV: payment_status = "VALIDATED"
  - النظام ينشئ معاملة مالية للوكالة (نوع: PV_PAYMENT، الاتجاه: IN)
- النظام يُرسل إشعار للـ Agent بتأكيد الدفع
- النظام يُعيد توجيه Gestionnaire للصفحة مع رسالة نجاح
- العملية تنتهي بنجاح

### 3.3 إطلاق الأموال للفنانين - تدفق العملية المعقدة جداً

- Gestionnaire يبدأ العملية بالدخول إلى صفحة PVs
- Gestionnaire يفتح PV مغلق ومدفوع (payment_status = VALIDATED)
- Gestionnaire يرى تفاصيل PV: المبلغ الإجمالي، المبلغ المدفوع، الأموال المستحقة لكل فنان
- Gestionnaire يضغط على زر "إطلاق الأموال"
- النظام يتحقق من أن PV مدفوع (payment_status = VALIDATED)
- إذا لم يكن مدفوعاً: النظام يعرض رسالة خطأ "يجب أن يتم تأكيد الدفع أولاً"، العملية تنتهي
- إذا كان مدفوعاً: النظام يتحقق من أن الأموال لم تُطلق بعد (funds_released_at = null)
- إذا أُطلقت الأموال بالفعل: النظام يعرض رسالة خطأ "تم إطلاق الأموال بالفعل"، العملية تنتهي
- إذا لم تُطلق: النظام يتحقق من أن المبلغ المدفوع يساوي المبلغ الإجمالي بالضبط
- إذا لم يتطابق: النظام يعرض رسالة خطأ "المبلغ المدفوع يجب أن يساوي المبلغ الإجمالي بالضبط"، العملية تنتهي
- إذا تطابق: النظام يحسب المبالغ المستحقة لكل فنان بناءً على استخدام أعماله في PV
- النظام يحصل على محفظة الوكالة
- النظام يتحقق من أن رصيد محفظة الوكالة >= المبلغ الإجمالي
- إذا لم يكن الرصيد كافياً: النظام يعرض رسالة خطأ "رصيد محفظة الوكالة غير كافٍ"، العملية تنتهي
- إذا كان الرصيد كافياً: النظام يبدأ معاملة مالية (Transaction)
  - النظام يخصم من محفظة الوكالة المبلغ الإجمالي
  - لكل فنان لديه أعمال في PV:
    - النظام يحصل على محفظة الفنان (أو ينشئها إذا لم تكن موجودة)
    - النظام يضيف المبلغ المستحق لمحفظة الفنان
    - النظام ينشئ معاملة مالية للفنان (نوع: PV_PAYMENT، المبلغ: موجب)
    - النظام ينشئ معاملة مالية للوكالة (نوع: PV_PAYMENT، الاتجاه: OUT)
    - النظام يُرسل إشعار للفنان بإطلاق الأموال
  - النظام يحدث PV: funds_released_at = الوقت الحالي
  - إذا كان مرتبطاً بمهمة: النظام يغير حالة المهمة إلى "DONE"
- النظام يُعيد توجيه Gestionnaire للصفحة مع رسالة نجاح
- العملية تنتهي بنجاح

## 4. Admin (المدير العام)

### 4.1 مراجعة وموافقة تسجيل Artist - تدفق العملية المعقدة

- Admin يبدأ العملية بالدخول إلى صفحة إدارة المستخدمين
- Admin يرى قائمة الفنانين قيد المراجعة (PENDING_VALIDATION) في وكالته
- Admin يفتح ملف فنان معين
- Admin يراجع المعلومات الشخصية ووثيقة الهوية
- Admin يختار إما الموافقة أو الرفض
- إذا اختار الموافقة:
  - Admin يضغط على زر "موافقة"
  - النظام يبدأ معاملة قاعدة بيانات (Database Transaction)
    - النظام يغير حالة الفنان إلى "APPROVED"
    - النظام يربط الفنان بالوكالة
    - النظام ينشئ محفظة للفنان (أو يستخدم المحفظة الموجودة)
    - النظام يربط المحفظة بالفنان
  - النظام يُرسل إيميل للفنان بالموافقة
  - النظام يُرسل إشعار للفنان بالموافقة
  - النظام يُعيد توجيه Admin لصفحة إدارة المستخدمين مع رسالة نجاح
  - العملية تنتهي بنجاح
- إذا اختار الرفض:
  - Admin يضغط على زر "رفض"
  - النظام يعرض حقل لإدخال سبب الرفض
  - Admin يدخل سبب الرفض (على الأقل 10 أحرف)
  - Admin يضغط على زر "تأكيد الرفض"
  - النظام يتحقق من أن سبب الرفض على الأقل 10 أحرف
  - إذا كان السبب أقل من 10 أحرف: النظام يعرض رسالة خطأ، Admin يعود لإدخال سبب أطول، العملية تعود إلى خطوة إدخال سبب الرفض
  - إذا كان السبب صحيحاً: النظام يُرسل إيميل للفنان بالرفض مع السبب
  - النظام يبدأ معاملة قاعدة بيانات (Database Transaction)
    - النظام يحذف وثيقة الهوية من الملفات (إذا كانت موجودة)
    - النظام يحذف المحفظة إذا كانت موجودة
    - النظام يحذف الفنان بشكل نهائي (Hard Delete)
    - النظام يحذف المستخدم بشكل نهائي (Hard Delete)
  - النظام يُعيد توجيه Admin لصفحة إدارة المستخدمين مع رسالة نجاح
  - العملية تنتهي بنجاح

## 5. Super Admin (المدير الأعلى)

### 5.1 تعيين Admin لوكالة - تدفق العملية

- Super Admin يبدأ العملية بالدخول إلى صفحة الوكالات
- Super Admin يفتح وكالة معينة
- Super Admin يضغط على زر "تعيين Admin"
- Super Admin يختار Admin موجود من القائمة أو ينشئ Admin جديد
- إذا اختار Admin موجود:
  - النظام يتحقق من أن الوكالة ليس لديها Admin (agency.admin_id = null)
  - إذا كان لدى الوكالة Admin: النظام يعرض رسالة خطأ "هذه الوكالة لديها Admin بالفعل، يرجى إزالة Admin الحالي أولاً"، العملية تنتهي
  - إذا لم يكن لدى الوكالة Admin: النظام يربط Admin بالوكالة
  - النظام يحدث agency.admin_id
  - النظام يُرسل إيميل للـ Admin بإشعار التعيين
  - النظام يُعيد توجيه Super Admin لصفحة الوكالة مع رسالة نجاح
  - العملية تنتهي بنجاح
- إذا اختار إنشاء Admin جديد:
  - Super Admin يملأ النموذج: الاسم، البريد الإلكتروني، كلمة المرور، رقم الهاتف
  - Super Admin يضغط على زر "إنشاء"
  - النظام يبدأ التحقق من البيانات: يتحقق من وجود الاسم، يتحقق من صحة البريد الإلكتروني، يتحقق من قوة كلمة المرور
  - النظام يتحقق من أن البريد الإلكتروني غير مستخدم
  - إذا كان البريد مستخدماً: النظام يعرض رسالة خطأ "البريد الإلكتروني مستخدم بالفعل"، Super Admin يعود لإدخال بريد آخر، العملية تعود إلى خطوة ملء النموذج
  - إذا كان البريد غير مستخدم: النظام يتحقق من أن الوكالة ليس لديها Admin
  - إذا كان لدى الوكالة Admin: النظام يعرض رسالة خطأ "هذه الوكالة لديها Admin بالفعل"، العملية تنتهي
  - إذا لم يكن لدى الوكالة Admin: النظام ينشئ مستخدم جديد
  - النظام يُعيّن دور "admin" للمستخدم
  - النظام يربط المستخدم بالوكالة
  - النظام يحدث agency.admin_id
  - النظام يُرسل إيميل للـ Admin الجديد
  - النظام يُعيد توجيه Super Admin لصفحة الوكالة مع رسالة نجاح
  - العملية تنتهي بنجاح

## ملاحظات عامة

- جميع العمليات تخضع للتحقق من الصلاحيات قبل البدء
- جميع المعاملات المالية تُنفذ داخل معاملات قاعدة بيانات (Database Transactions) لضمان السلامة
- في حالة الخطأ، تُوقف العملية ويُعرض رسالة خطأ واضحة
- جميع التحققات (Validations) تُنفذ قبل حفظ البيانات
- عند إرسال إشعارات، تصل للمستخدمين المعنيين فوراً
- جميع العمليات تُسجل في سجل المعاملات (Transactions) للتتبع
- في العمليات المعقدة، يتم التحقق من كل شرط على حدة قبل المتابعة
